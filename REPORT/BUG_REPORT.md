# Bug Report — ZHELEZO Landing

Дата: 2026-07-18. Полный ручной проход по всем 13 файлам `Landing/src/**` в этой сессии
(предыдущие проходы уже проверили XSS/дубли/битые ссылки — см. `SECURITY_REPORT.md`, не
повторяю здесь).

## Найденные и исправленные баги

### 1. `LuckyWheel.tsx` — `window.setTimeout` без очистки при размонтировании
**Серьёзность**: средняя (сейчас низкий практический риск, реальный риск при будущих
изменениях). **Категория**: race condition / потенциальная утечка.

После клика "Вращать" ставился `window.setTimeout(..., 4200)`, обновляющий состояние
(`setIsSpinning`, `setResult`) без сохранения ID таймера и без очистки в cleanup-функции
эффекта. Если компонент размонтируется раньше 4.2с — `setState` на размонтированном
компоненте (React-предупреждение в консоли, в проектах посерьёзнее — реальная утечка).

**Почему низкий риск именно сейчас**: в приложении нет роутинга (`App.tsx` — статический
рендер, `LuckyWheel` монтируется один раз и никогда не убирается условно), так что
практически component не размонтируется в текущей версии сайта. Но это ровно тот паттерн
багов, который явно просили искать ("race condition", "утечки памяти") — и он реален с точки
зрения корректности кода, будущие изменения (например, условный рендер секции) немедленно
проявили бы проблему.

**Исправление**: ID таймера сохраняется в `useRef`, очищается в `useEffect` cleanup.
Коммит `589fe3b`.

### 2. `Footer.tsx` — "мёртвые" ссылки соцсетей без видимой обратной связи
**Серьёзность**: низкая (UX), пограничная с accessibility. **Категория**: неудобство UX,
частично — некорректная доступность.

Иконки Twitter/Telegram/GitHub были `<a href="#">` с `onClick={e => e.preventDefault()}` —
для зрячих пользователей выглядели как рабочие ссылки, реагирующие на hover, но по клику
ничего не происходило без объяснения. Состояние "скоро" было закодировано только в
`aria-label`, невидимом для зрячих пользователей.

**Исправление**: заменено на `<button disabled>` — стандартный HTML-паттерн для "элемент
существует, но недоступен": видимое приглушение (`opacity-60`, `cursor-not-allowed`) для
зрячих + корректная disabled-семантика для ассистивных технологий. Коммит `05b0d80`.

### 3. Отсутствие ErrorBoundary — единая точка отказа всего UI
**Серьёзность**: средняя (архитектурная, не баг в текущем поведении). **Категория**:
обработка ошибок / надёжность.

Ни один компонент дерева не был обёрнут в `ErrorBoundary`. Любое необработанное исключение
при рендере (в текущем коде такого не найдено, но это защита от будущих регрессий) обрушило
бы всё React-дерево в пустой белый экран — на партнёрском лендинге это значит, что посетитель
теряет единственный путь к реферальной ссылке молча, без единого шанса восстановиться.

**Исправление**: добавлен `ErrorBoundary` (класс-компонент, `componentDidCatch`) с
минимальным fallback-UI, сохраняющим рабочую реферальную ссылку, плюс `window.onerror`/
`unhandledrejection` листенеры — оба репортят через существующий `trackEvent()` в GA4/
Яндекс.Метрику как новое событие `client_error`. Коммит `80c9579`.

## Не баг, но требует решения пользователя: `SocialProof.tsx`

**Не отношу это к "багам" технически** — данные в компоненте (`wins` — 5 захардкоженных
"живых" выигрышей, циклически ротируемых каждые 4с) полностью синтетические, не связаны с
реальными событиями на BC.Game. Компонент был явно оставлен предыдущей сессией как "реальный
CRO-элемент" (см. `Память/decisions.md`, запись про SocialProof) — то есть это осознанное
маркетинговое решение, не недосмотр.

Отмечаю это здесь, а не молча правлю, по двум причинам:
1. GEMINI.md (п.8) явно требует избегать "прямого обмана" в решениях по этому проекту —
   полностью фабрикованная "live"-активность (не помеченная как иллюстративная) в некоторых
   юрисдикциях/рекламных политиках может квалифицироваться как вводящая в заблуждение реклама.
2. Это продуктовое/юридическое решение, а не код-баг — не мне единолично решать, убирать
   компонент или оставлять.

**Рекомендация** (не исполнено, ждёт решения): либо явно и видимо пометить блок как
иллюстративный пример (например, мелкая подпись "иллюстративный пример" рядом с виджетом),
либо, если/когда у BC.Game появится публичный API реальных выигрышей, подключить настоящие
данные вместо захардкоженных.
