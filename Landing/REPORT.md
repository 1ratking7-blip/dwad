# Аудит и доработка zhelezo.space — отчёт

Дата: 2026-07-18
Объём: `Landing/` (React + TypeScript + Vite + Tailwind, статический сайт на Cloudflare Pages,
без бэкенда и БД). Все пункты ниже проверены на живом проде (https://www.zhelezo.space/),
не только локально.

## Контекст: почему часть пунктов ТЗ не применима

Сайт — полностью статический SPA без сервера, API и базы данных (нет `functions/`,
нет бэкенд-роутов). Поэтому по факту не применимы: SQL Injection, SSRF, RCE, аутентификация/
авторизация, оптимизация запросов к БД — эксплуатировать/чинить там нечего. Ниже — то, что
реально применимо и было проверено.

Также важно: в репозитории уже был коммит `39e386b` ("Full SEO/CRO/perf/a11y/security audit")
от предыдущей сессии, который поднял SEO 91→100, Accessibility 71→100, Performance 56→73.
Этот отчёт — второй проход поверх того аудита, поэтому находок по XSS/секретам/доступности
почти нет: та работа уже была сделана хорошо.

## Что исправлено (по коммитам)

1. **HSTS отсутствовал** (`377c754`) — заголовки уже включали CSP/X-Frame-Options/
   Permissions-Policy/Referrer-Policy, но не Strict-Transport-Security. Добавлен
   `max-age=63072000; includeSubDomains; preload`.

2. **CVE в зависимостях** (`a564bcc`) — `npm audit` нашёл moderate/high в vite/esbuild
   (path traversal и `server.fs.deny` bypass в **dev-сервере** — на прод, где раздаются
   только статические файлы из `dist/`, эти CVE не влияют). Патч без мажорного апгрейда
   не существовал, поэтому подняли vite 5→8 и `@vitejs/plugin-react`. Сборка и CSP-хэши
   инлайн-скриптов проверены — не изменились. `npm audit`: 0 уязвимостей.

3. **Производительность — код-сплиттинг** (`7ccecff`) — секции ниже первого экрана
   (LuckyWheel, Games, HowItWorks, FAQ, Footer) переведены на `React.lazy`/`Suspense`.
   Главный чанк уменьшился с ~298 КБ до ~272 КБ; framer-motion для LuckyWheel больше не
   тянется в критический путь рендера Hero.

4. **Производительность — блокирующий CSS шрифта** (`b76da24`) — `<link rel=stylesheet>`
   на Google Fonts блокировал первую отрисовку независимо от `display=swap` (swap влияет
   только на отрисовку текста конкретным шрифтом, а не на блокировку самого CSS-файла).
   Переведено на `preload` + переключение `rel` на `load` + `<noscript>`-фолбэк. Новый
   инлайн-скрипт захеширован и добавлен в CSP (`public/_headers`), проверено — консоль
   браузера чистая (0 ошибок в Lighthouse `errors-in-console`).

5. **Битый маршрут 404** (`abcef4b`) — сайт без клиентского роутера, но Cloudflare Pages
   по умолчанию отдавал `index.html` с кодом **200** на любой несуществующий путь
   (`/nonexistent-page-xyz` → 200). Добавлен `public/404.html` — теперь отдаётся настоящий
   404. Проверено на проде после деплоя.

Каждый пункт задеплоен на прод (`npm run deploy` → Cloudflare Pages) сразу после коммита,
не только закоммичен локально.

## Проверено и не исправлялось (не найдено проблем)

- **XSS / секреты**: прочитаны все компоненты (`src/**/*.tsx`) и `lib/*.ts`. Нет
  `dangerouslySetInnerHTML`, `eval`, хардкод-токенов/ключей. Все `target="_blank"` уже с
  `rel="noopener noreferrer"`. GA4/Яндекс.Метрика ID — публичные идентификаторы, не секреты.
- **Сборка/типы/линтер**: `tsc --strict` (с `noUnusedLocals`/`noUnusedParameters`) и
  `vite build` — чисто, без предупреждений. Отдельного ESLint в проекте нет и не было —
  не стал заводить с нуля посреди аудита: это отдельное решение по тулингу, а не баг.
- **Мёртвый код/дубли**: не найдено — реферальная ссылка уже вынесена в `lib/links.ts`,
  трекинг — в `lib/analytics.ts`, компоненты не дублируют друг друга.
- **Битые ссылки**: `/`, `/privacy-policy`, `/terms`, `/sitemap.xml`, `/robots.txt`, иконки —
  все 200. Внешняя реферальная ссылка на BC.Game сначала выглядела мёртвой (curl вернул 415),
  но это анти-бот фильтр партнёрской платформы по User-Agent — с браузерным UA ссылка
  отдаёт 200. Ложная тревога, реальной проблемы нет.
- **CORS**: `Access-Control-Allow-Origin: *` на статических ответах — это дефолт самого
  Cloudflare Pages для статики, не наша конфигурация. Риска нет: сайт без cookie-based
  сессий и приватных данных, которые можно было бы утащить кросс-origin.

## Lighthouse: честная оговорка

Гонял живой Lighthouse (headless Chrome) на www.zhelezo.space трижды: Performance
стабильно ~69-70, Accessibility 100, Best Practices 77, SEO 100 — как до, так и после
двух перф-фиксов (замеры почти не сдвинулись).

Разобрал трассировку: **~85% времени "Script Evaluation" (2.3 c из 2.4 c) помечено как
"Unattributable"**, а не привязано к нашему бандлу (у него самого — ~95 мс). Это
указывает, что узкое место — не наш JS, а особенности среды, где я гонял замер
(headless Chrome в контейнере/песочнице этой машины поверх ещё и синтетического CPU-
троттлинга Lighthouse — двойное замедление). Настоящий Performance-балл на реальном
мобильном устройстве или через **pagespeed.web.dev** (инфраструктура Google, откалиброванная
для таких замеров) с высокой вероятностью будет заметно выше 70. Заявить "Lighthouse ≥95
достигнут" на основании моих цифр было бы нечестно — рекомендую перепроверить на
pagespeed.web.dev и ориентироваться на те цифры.

Best Practices 77 почти целиком объясняется двумя пунктами, которые лежат вне нашего кода:
"third-party cookies" и "inspector issue: Cookie" — оба про cookies, которые ставит сам
Яндекс.Метрика (`mc.yandex.ru`). Убрать можно только вместе с самой Метрикой.

## Не исправлено — нужен доступ, которого у меня нет

**Голый домен `https://zhelezo.space` (без www) не открывается по HTTPS** — таймаут
соединения. По HTTP редирект на `www.zhelezo.space` работает (301, Namecheap URL Forward),
но по HTTPS Namecheap-редирект не обслуживает TLS для apex-домена вообще. Современные
браузеры (Chrome и т.п.) по умолчанию апгрейдят набранный `zhelezo.space` до `https://`
ещё до перехода — то есть пользователь, который наберёт домен без `www`, скорее всего
получит "не удаётся открыть страницу", а не редирект.

Исправить руками из этой сессии не могу — нет учётных данных Namecheap/Cloudflare DNS в
проекте (проверил `.env`, GEMINI.md, `Память/context.md` — токенов нет). Варианты фикса:
1. Перенести DNS-зону `zhelezo.space` на Cloudflare (раз хостинг уже там) и настроить
   redirect rule на уровне Cloudflare — у них apex получает нормальный TLS.
2. Включить PremiumDNS/SSL для форвардинга на стороне Namecheap, если такая опция есть
   на тарифе.

Если дадите доступ (API-токен Namecheap или включите Cloudflare как DNS-провайдера зоны) —
доделаю в следующей сессии.

## Итог

5 реальных фиксов задеплоены на прод и закоммичены отдельными коммитами. Критических
уязвимостей (XSS/секреты/CSRF-поверхность) не найдено — предыдущий аудит уже закрыл
основное. Главная оставшаяся проблема — HTTPS на голом домене, требует доступа к DNS.
Performance-цифры из моего локального Lighthouse не считаю авторитетными — просьба
свериться с pagespeed.web.dev.
