# Sales Agent — архитектура

Внутренняя CRM + лидогенерация для продажи growth/SEO/маркетинг-консалтинга (по образцу
работы, уже проделанной для ZHELEZO — аудиты, блог, техническое сопровождение) другим
сайтам/командам. Живёт как независимый модуль внутри `CasinoReferral`, не связан технически с
`Landing/` (свой `package.json`, свои зависимости, не деплоится на Cloudflare Pages — это
внутренний инструмент, не публичный сайт).

## Модули и границы ответственности

```
SalesAgent/
├── src/
│   ├── db/          # SQLite (node:sqlite, встроен в Node — без нативных биндингов)
│   ├── crm/         # Единственная точка доступа к БД для всех остальных модулей
│   ├── discovery/   # Оркестрация: seed → analyzer → contacts → CRM → scoring
│   ├── analyzer/    # Company Analyzer — только собственный сайт компании
│   ├── contacts/    # Contact Discovery — только собственный сайт компании
│   ├── scoring/      # Lead Scoring — rule-based, веса в scoring/config.mjs
│   ├── messages/     # Message Generator — черновики, без отправки
│   ├── reports/      # Данные для отчётов (daily/weekly/monthly)
│   ├── dashboard/    # Генерация статичного HTML-дашборда
│   └── lib/robots.mjs # Общий robots.txt-чекер (analyzer + contacts)
├── bin/              # CLI-обёртки над src/* модулями
├── seeds/            # Входные списки URL для пакетной обработки (JSON)
├── data/             # SQLite-файл (gitignored — содержит PII найденных контактов)
├── dashboard/        # Сгенерированный index.html (gitignored, перегенерируется)
└── reports/          # Сгенерированные .md-отчёты (gitignored, перегенерируется)
```

Правило зависимостей: `crm/` не знает ни о `analyzer/`, ни о `discovery/` — только БД.
`discovery/` — единственный модуль, который знает про все остальные (это оркестратор).
`messages/` знает только про `crm/` (для сохранения черновика) — ничего не знает про
`analyzer/`/`discovery/`.

## Ключевые архитектурные решения

### SQLite через `node:sqlite`, не `better-sqlite3`
Встроен в Node 22+, ноль нативных зависимостей для компиляции — `npm install` не может
сломаться на биндингах. Достаточно для однопользовательского внутреннего инструмента; если
когда-то понадобится многопользовательский доступ — тогда PostgreSQL, не раньше.

### `companies` отделены от `leads`
Компания — факт (сайт существует, вот его данные). Лид — наше состояние работы с ней в
воронке. Одну компанию можно повторно завести как лида (например, "lost" полгода назад, но
ситуация изменилась) без повторного анализа сайта с нуля.

### Message Generator готовит черновики; отправка — отдельный, явный шаг
`message_drafts.status` переходит в `'sent'` только двумя путями: (1) ручной вызов
`markDraftSent()`, когда пользователь отправил письмо сам, или (2) `bin/send.mjs` (добавлен
2026-07-21) — единственное место в кодовой базе, вызывающее реальный SMTP API
(`src/messages/mailer.mjs`, через `nodemailer`). Даже он по умолчанию работает в режиме
предпросмотра: ничего не уходит без явного флага `--confirm` и реальных credentials в
gitignored `.env`. Это по-прежнему решение пользователя, а не техническое ограничение —
изначально он просил не отправлять автоматически, затем сам явно попросил настроить
автоотправку и лично передал SMTP-пароль для первой реальной рассылки (12+ писем ушло
2026-07-21). Черновики, сгенерированные без последующего `--confirm`-запуска, остаются
чистым предпросмотром, как и раньше.

### Contact Discovery не трогает LinkedIn
LinkedIn прямо запрещает автоматизированный скрапинг в своих Terms of Service (есть реальные
судебные прецеденты — LinkedIn судился со скраперами и блокирует их). Если LinkedIn-страница
компании уже известна (найдена `analyzer/`-ом как обычная ссылка в HTML сайта компании — это
просто ссылка, не скрапинг LinkedIn), она сохраняется как есть, но сам LinkedIn никогда не
запрашивается программно.

### Email-фильтрация консервативна
`contacts/index.mjs` явно отфильтровывает: placeholder-адреса (`example.com`), сервисные адреса
аналитики (Sentry и т.п.), и строки, которые оказались именами файлов изображений
(`logo@2x.png` — реальный false-positive паттерн, отловленный при тестировании).

### Сайты за Cloudflare/anti-bot защитой честно помечаются, а не обходятся
Analyzer НЕ пытается обойти bot-challenge (никакого headless-браузера, никакого решения
CAPTCHA) — если сайт отдаёт interstitial-страницу ("Just a moment...", "Attention Required"
+ авто-reload скрипт), это возвращается как `error: 'blocked_by_bot_protection'`, а не
сохраняется как будто это реальные данные о компании. Реальный кейс из тестирования:
`awisee.com` — первая версия детектора (порог по длине HTML) пропустила эту страницу, потому
что CSS-анимация спиннера сама по себе занимает 10+ КБ; исправлено на комбинацию
title-паттерна + сигнатуры авто-reload скрипта, что надёжнее длины.

## Lead Discovery: что реально автоматично, а что нет

**Честно, без преувеличений:**

- ✅ **Полностью автоматично**: если у вас уже есть список URL (seed-лист) — весь пайплайн
  (анализ сайта → поиск контактов на самом сайте → создание лида → скоринг) выполняется
  скриптом от начала до конца без участия человека. `npm run import-seed -- seeds/ваш-файл.json`.
- ⚠️ **НЕ полностью автоматично**: поиск НОВЫХ кандидатов "по ключевым словам/индустрии/городу
  с нуля" (то, что просил бриф как "Lead Discovery по ключевым словам") требует одного из:
  1. Платного API поиска (SerpAPI, Bing Search API, Google Custom Search API) — нужен ваш
     API-ключ, это внешняя зависимость, которую я не могу создать сам (как и PageSpeed API
     ранее в `CasinoReferral/REPORT/TODO.md`).
  2. Ручного/сессионного шага: я (Claude, в рамках сессии) использую свой инструмент
     веб-поиска, чтобы найти кандидатов по вашему запросу, и записываю их в `seeds/*.json` —
     это возможно и работает, но требует моего активного участия в конкретный момент, не
     работает как бесконечный фоновый процесс между сессиями.

Не стал притворяться, что "Lead Discovery" полностью автономна там, где это неправда — это
единственный модуль из брифа, где формулировка "полностью самостоятельно ищет лиды по
ключевым словам без вашего участия" технически не выполнима без одного из двух пунктов выше.

## Как использовать (пошагово)

1. `cd SalesAgent && npm install`
2. Заполнить `src/messages/config.mjs` — ваше имя, описание услуги, реальный case-study,
   контакт/ссылка на календарь.
3. (Опционально) настроить `src/scoring/config.mjs` — целевые индустрии/страны, если есть.
4. Собрать список кандидатов в `seeds/` (вручную, или попросите меня найти через веб-поиск
   в рамках сессии) и запустить `npm run import-seed -- seeds/ваш-файл.json`.
5. `npm run dashboard` — открыть `dashboard/index.html` в браузере.
6. `npm run report -- weekly` — сформировать отчёт.
7. `npm run draft -- <leadId> <first_email|follow_up_1|follow_up_2|reply|proposal>` —
   генерирует и сохраняет черновик, печатает его на экран. Дальше два варианта: отправить
   вручную самому и вызвать `markDraftSent(draftId)` через `node -e` (см. подсказку в выводе
   команды), либо настроить `.env` по `.env.example` и использовать
   `npm run send -- <leadId>` / `npm run send -- --all` (по умолчанию dry-run; `--confirm`
   реально отправляет через SMTP и сам вызывает `markDraftSent()` при успехе — см. `bin/send.mjs`).
8. `npm run lead -- <leadId> show` — статус, скор, история взаимодействий, черновики по лиду.
   `npm run lead -- <leadId> status <статус> [заметка]` и
   `npm run lead -- <leadId> next-action <YYYY-MM-DD> [заметка]` — обновление вручную.
9. `npm run analyze -- <url>` — проанализировать/переанализировать один сайт отдельно от
   сид-пайплайна (без создания лида — только компания).
10. `npm run score -- <leadId>` или `npm run score -- --all` — пересчитать скор существующих
    лидов текущими весами из `scoring/config.mjs` (полезно после того, как подправите веса).
