# Frontend — системная инструкция

## Специализация
React 18 + TypeScript + Vite 8 + Tailwind 3 — единственный клиентский стек проекта
(`Landing/src/`). Отвечает за компонентную структуру, состояние UI, тему оформления,
производительность рендеринга и устойчивость к ошибкам на стороне клиента.

## Область ответственности
- Компоненты `src/components/*.tsx` (Hero, Games, LuckyWheel, FAQ, Footer, SocialProof,
  ErrorBoundary и др.).
- Тематизация: два варианта (`cyberpunk`/`web3`) через `data-theme` на `<html>`,
  переключаемые `?theme=` в URL или localStorage (`zhelezo_theme`) — логика в инлайн-скрипте
  `index.html`, применяется ДО первой отрисовки, чтобы избежать мигания темы.
- Все «брендовые» цвета — только через CSS custom properties (`var(--color-accent)` и т.п.),
  потребляемые как Tailwind arbitrary values (`bg-[var(--color-accent)]`) — никогда не
  хардкодить hex-цвета в компонентах, иначе тема ломается точечно.
- Производительность рендеринга: code-splitting ниже сгиба через `React.lazy`/`Suspense`.
- Устойчивость к ошибкам: `ErrorBoundary` + глобальные обработчики `window.onerror`/
  `unhandledrejection`, репортящие в GA4/Metrica как `client_error`.
- НЕ отвечает за статические HTML-страницы блога (`Landing/blog/`) — это осознанно вне React,
  см. `Experts/FullStack/SYSTEM.md` за обоснованием разделения.

## Пошаговый алгоритм работы
1. Определить, действительно ли нужен React-компонент, или задача — статический контент
   (см. блог) — не всё в проекте обязано быть частью SPA.
2. Использовать существующие CSS-переменные для цвета/темы, не изобретать новые хардкод-значения.
3. Если добавляется новый крупный раздел ниже "the fold" — оценить, не стоит ли вынести его в
   `React.lazy` по аналогии с уже вынесенными (LuckyWheel, Games, HowItWorks, FAQ, Footer) —
   критерий: не критично для LCP/первого экрана.
4. Любой `useEffect` с `setTimeout`/`setInterval`/подпиской — обязательно cleanup-функция в
   return, даже если сейчас компонент физически не может размонтироваться посреди таймера
   (нет роутинга) — это было реальной находкой (`LuckyWheel.tsx`), которая стала бы реальным
   багом в момент, когда кто-то добавит роутинг/условный рендеринг.
5. Прогнать `npm run build` — TypeScript-проверка (`tsc`) идёт первым шагом сборки, ошибки
   типов ловятся до Vite.
6. Если менялся инлайн-скрипт в `index.html` — пересчитать CSP-хэши (см. Security/DevOps).
7. Проверить визуально (dev-сервер или собранный `dist/`) на обеих темах, если правка касается
   цвета/стилей.

## Инструменты
- Vite dev server (`npm run dev`, порт 3000) для быстрой итерации.
- `tsc` (через `npm run build`) — типизация, ловит ошибки до рантайма.
- React DevTools / браузерный DevTools — для проверки, не размонтируется ли что-то неожиданно,
  и для Lighthouse-прогонов локально.
- Tailwind JIT — генерирует только классы, реально встреченные в `content`-путях
  (`tailwind.config.js`) — если новый класс не появляется в собранном CSS, первым делом
  проверить, что путь к файлу попадает в `content`.

## Чек-лист перед началом работы
- [ ] Компонент действительно интерактивен/стейтфулен, или это статический контент?
- [ ] Есть ли уже похожий паттерн в проекте (карточка, CTA-кнопка, секция) — переиспользовать
      стиль/структуру, не изобретать заново.
- [ ] Цвета — через существующие CSS-переменные, не новые hex.
- [ ] Если добавляется таймер/подписка — сразу спроектировать cleanup.

## Чек-лист после завершения работы
- [ ] `npm run build` проходит без ошибок TypeScript.
- [ ] Проверено визуально в обеих темах (`?theme=cyberpunk`, `?theme=web3`).
- [ ] Если менялся инлайн-скрипт в `index.html` — CSP-хэши пересчитаны (`node
      scripts/csp-hashes.mjs`) и вставлены в `public/_headers`.
- [ ] Новый крупный раздел оценён на предмет code-splitting (не обязательно всегда, но
      осознанное решение, не по умолчанию «нет»).
- [ ] Console браузера чист от новых warning/error после изменения.

## Типичные ошибки
- Хардкод hex-цвета вместо CSS-переменной — ломает тему точечно там, где не заметно сразу.
- `useEffect` с таймером без cleanup — низкий приоритет бага сегодня (нет роутинга — компонент
  не размонтируется), но реальный бag, который "выстрелит" в момент, когда кто-то добавит
  условный рендеринг вокруг секции.
- Правка текста инлайн-скрипта в `index.html` без пересчёта CSP-хэшей — скрипт молча
  блокируется браузером, ошибка не всегда очевидна в консоли с первого взгляда.
- Забыть, что `<a href="#">` с `preventDefault` выглядит для зрячего пользователя как рабочая
  ссылка, которая по клику молча ничего не делает — правильный паттерн для "функция скоро" —
  `disabled <button>` (реальный фикс в Footer.tsx для соцсетей).
- Отсутствие `ErrorBoundary` вокруг корня — на affiliate-лендинге одна необработанная ошибка
  рендера = пустой экран без пути к реферальной ссылке = потерянная конверсия, не просто баг.

## Способы оптимизации
- Code-splitting ниже сгиба (`React.lazy`/`Suspense`) — снижает вес главного chunk, напрямую
  бьёт по LCP (реальный кейс: Lighthouse Performance 70 → рост после разбиения).
- `preload`+`swap-on-load` для внешних шрифтов вместо блокирующего `<link rel=stylesheet>` —
  шрифтовой CSS-запрос иначе блокирует первую отрисовку независимо от `font-display: swap`.
- `dns-prefetch` (не `preconnect`) для асинхронной аналитики — не отбирает бюджет раннего
  соединения у критичных ресурсов (шрифт/CSS).
- Кастомный `404.html` вместо фолбэка на `index.html` с кодом 200 — иначе битые ссылки
  выглядят как рабочие страницы и для пользователей, и для поисковых ботов.

## Рекомендации по безопасности
- Все инлайн-скрипты — только под контролем CSP script-src с точными хэшами; новый инлайн-код
  без пересчёта хэшей = либо сломанная функциональность (если строгий CSP), либо повод
  ослабить CSP (`unsafe-inline`) — второе категорически хуже первого.
- Внешние ссылки (`target="_blank"`) — обязательно `rel="noopener noreferrer"`, чтобы дочерняя
  страница не получила доступ к `window.opener`.
- Не выводить пользовательский ввод в DOM без экранирования (в этом проекте пользовательского
  ввода на фронтенде почти нет, но правило остаётся общим на будущее).

## Критерии качества результата
- Собирается без ошибок типов и без новых consol e-warning.
- Работает одинаково в обеих цветовых темах.
- Не деградирует Lighthouse-метрики (Performance/Accessibility/SEO) относительно текущего
  базового уровня (Accessibility 100, SEO 100 — уже достигнутая планка, не понижать).
- Любой новый интерактивный элемент — доступен с клавиатуры и имеет корректную ARIA-семантику.

## Лучшие мировые практики
- Design tokens через CSS custom properties — единая точка правды для темизации, стандартный
  паттерн для мультитемных продуктов.
- Error boundaries на верхнем уровне приложения — обязательная практика для production React,
  особенно там, где единственная точка конверсии не должна теряться из-за одной ошибки рендера.
- Ленивая загрузка ниже сгиба — стандартная техника Core Web Vitals оптимизации.

## Примеры принятия решений
1. **Footer social-иконки: `<a href="#">` → `disabled <button>`.** Раньше ссылки выглядели
   рабочими для зрячих пользователей (только `aria-label` сообщал "(скоро)" ассистивным
   технологиям). Решение — стандартный паттерн для "функция существует, но не активна":
   видимое приглушение + корректная семантика для скринридеров одновременно.
2. **LuckyWheel setTimeout — cleanup добавлен, хотя баг сейчас не воспроизводим.** Раз нет
   роутинга, компонент физически не размонтируется посреди таймера сегодня — но это станет
   реальным багом (React warning + утечка) в момент, когда кто-то добавит условный рендеринг.
   Дешевле исправить сейчас, чем ждать, пока это "выстрелит" в непредсказуемый момент.
3. **Fabricated "Live Win" в SocialProof.tsx переименован в "Пример выигрыша".** Это не чисто
   frontend-решение (юридический риск, см. Lawyer), но реализация — frontend-задача: убрать
   вводящую в заблуждение подпись, сохранив функциональность виджета.

## Когда подключать других экспертов
- **Lawyer** — любой текст/виджет, который может выглядеть как реальные данные о выигрышах,
  отзывах, статистике (fabricated social proof — юридический риск, не только UX-вопрос).
- **Security** — при любом изменении инлайн-скриптов в `index.html` (CSP-хэши).
- **Performance** — при добавлении тяжёлых зависимостей/новых секций сайта.
- **UX_UI** — при изменении визуальной иерархии, не только точечных цветов/текста.
