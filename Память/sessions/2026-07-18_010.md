# Сессия 2026-07-18 (010) — SalesAgent: B2B CRM для лидогенерации

## Контекст
Пользователь запросил "AI Sales Agent 24/7" — полноценную систему поиска клиентов/CRM/
скоринга/черновиков писем, работающую максимально автономно. Прежде чем строить —
уточнил критичное: для какого бизнеса ищем клиентов (Message Generator не может писать питчи,
не зная продукт) и где живёт проект. Ответ: продажа growth/SEO-консалтинга (по образцу работы
над ZHELEZO) другим сайтам, модуль внутри `CasinoReferral`.

## Архитектурное решение
Новый независимый модуль `SalesAgent/` — свой `package.json`, не связан технически с
`Landing/`, не деплоится на Cloudflare Pages (внутренний инструмент, не публичный сайт).
SQLite через встроенный `node:sqlite` (Node 22+), не `better-sqlite3` — ноль нативных
зависимостей для компиляции.

## Реализовано (все 8 модулей из брифа)
1. **DB + CRM core** (`src/db/`, `src/crm/`) — схема (companies/contacts/leads/interactions/
   message_drafts), CRUD-слой, через который проходят все остальные модули.
2. **Company Analyzer** (`src/analyzer/`) — фетчит ТОЛЬКО собственный сайт компании
   (robots.txt-проверка обязательна), извлекает название/описание/соцсети/tech-stack
   (WordPress/Shopify/Next.js/GA и т.п. по сигнатурам в HTML). Протестировано на реальном
   сайте (wordpress.org) — корректно определил все поля.
3. **Contact Discovery** (`src/contacts/`) — email/форма обратной связи ТОЛЬКО со страницы,
   которую нашёл Analyzer на сайте самой компании. Явный отказ от скрапинга LinkedIn (прямое
   нарушение их ToS, реальные судебные прецеденты). Email-фильтр консервативен — отсеивает
   placeholder-адреса и false positives вида `logo@2x.png` (протестировано).
4. **Lead Scoring** (`src/scoring/`) — rule-based, веса в отдельном конфиге, честно объяснено
   в документации, почему rule-based, а не ML (нет исторических данных о конверсии).
5. **Message Generator** (`src/messages/`) — черновики only, `markDraftSent()` — единственное
   место, где статус становится 'sent', и только по ручному вызову. Ключевая деталь:
   `insight.mjs` генерирует ОДНО честное наблюдение из реально проанализированных данных
   (не выдуманный комплимент) — тот же анти-фабрикация принцип, что и весь проект.
6. **Dashboard + Reports** (`src/dashboard/`, `src/reports/`) — статичный HTML (тёмная тема,
   в стиле ZHELEZO), markdown-отчёты daily/weekly/monthly.
7. **Lead Discovery** (`src/discovery/`) — полный пайплайн seed→analyze→contacts→CRM→scoring
   автоматичен для готового списка URL. Честно задокументировано в `docs/ARCHITECTURE.md`:
   поиск НОВЫХ кандидатов "по ключевым словам с нуля" не полностью автономен — нужен либо
   платный SERP API (внешняя зависимость, ключ от пользователя), либо мой веб-поиск в рамках
   сессии. Не стал изображать, что эта часть брифа выполнима "без участия пользователя",
   когда технически это не так.

## Проверено
Полный pipeline протестирован end-to-end несколько раз на реальных сайтах (wordpress.org,
nodejs.org) — analyze → CRM → score → draft, каждый раз с очисткой тестовых данных перед
финальным коммитом (`data/sales_agent.db` не коммитится, gitignored).

## Что нужно от пользователя
Заполнить `src/messages/config.mjs` (реальное имя, услуга, case study, контакт) перед
использованием — сейчас там плейсхолдеры. Собрать первый seed-лист (или попросить меня найти
кандидатов через веб-поиск в следующей сессии).

## Коммиты
Ожидает коммита/пуша на момент записи.
