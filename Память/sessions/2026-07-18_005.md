# Сессия 2026-07-18 (005) — блог-раздел (MVP: 2 статьи)

## Контекст
Из плана (`REPORT/ACTION_PLAN.md`/`GROWTH_PLAN.md`) пользователь выбрал следующим шагом
блог-раздел — самый крупный пункт, ранее отложенный до явного решения по объёму (одностраничник
физически не может расти в SEO дальше без множества страниц контента, подтверждено реальным
конкурентным анализом в `Analytics/COMPETITORS.md`).

## Архитектурное решение
Блог — статический многостраничный Vite-билд (`Landing/blog/`), не React SPA-роутинг:
- Каждая статья — отдельный `blog/<slug>/index.html`, чистый HTML/Tailwind, без React/гидратации.
- Стили переиспользуются через `blog/blog-entry.ts` (`import '../src/index.css'`) — Vite
  извлекает тот же CSS-бандл, что и у главной страницы, без дублирования.
- `vite.config.ts` → `build.rollupOptions.input` — явные entry-points на каждую страницу (не
  glob) — намеренно, задокументировано в `blog/README.md`, что при добавлении статьи нужно
  руками добавить путь сюда, иначе Vite её не соберёт.
- `tailwind.config.js` content расширен на `./blog/**/*.html`.

Причина отказа от React-роутинга: Bing/Yandex надёжнее индексируют статический HTML, чем
JS-рендеринг SPA; для блога не нужна интерактивность, только текст+ссылки — так что чистый HTML
меньше по весу и полностью crawlable без выполнения JS.

## Что сделано
1. `blog/index.html` — листинг статей.
2. `blog/bc-game-promo-code-2026/index.html` — под транзакционный запрос "BC.Game промокод"
   (подтверждён реальным поиском в `COMPETITORS.md` как то, что реально ищут). Честно объясняет,
   что отдельного текстового кода нет — бонус активируется через реферальную ссылку, без
   выдумывания несуществующего "промокода" (GEMINI.md п.8).
3. `blog/provably-fair-explained/index.html` — информационная статья, расширенная версия
   объяснения из `Texts/forum_bitcointalk.md`/`Texts/quora_answers.md` под формат полноценной
   статьи.
4. Внутренняя перелинковка: статьи ↔ `/#hero`, `/#games`, друг на друга; реф-ссылки с
   `subId1=seo&subId2=<slug>` (схема из `Analytics/HYPOTHESES.md`) для трекинга конверсии по
   конкретной статье.
5. `public/sitemap.xml` — добавлены 3 новых URL.
6. `Footer.tsx` — добавлена ссылка "Блог" (без неё раздел был бы orphan-страницами без
   внутренних ссылок с сайта).
7. Проверка билда: `npm run build` проходит, CSP-хэши (`node scripts/csp-hashes.mjs`)
   совпадают с уже прописанными в `public/_headers` (инлайн-скрипты в index.html не менялись,
   пересчёт не требовался). Отдельно проверено, что Tailwind реально сканирует новые
   `blog/**/*.html` файлы и генерирует использованные там классы (ложная тревога в процессе
   проверки — экранирование в моих же командах grep/node, не баг сборки).
