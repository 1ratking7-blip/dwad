# Сессия 2026-07-20 (002) — Полная мультиязычность (RU/EN/VI)

## Контекст
Пользователь запросил полноценную поддержку 3 языков на сайте (RU/EN/VI) — детальный технический
бриф: i18n-система, словари, никаких строк в компонентах, hreflang/canonical/локализованный
sitemap, переключатель языка с localStorage, автоопределение по браузеру, проверка на предмет
битых ссылок/смешения языков/дублей метатегов, коммит только если всё прошло успешно.

## Архитектурное решение
У проекта уже был готовый паттерн именно для этой задачи: статический multi-page блог уже строил
отдельный `index.html` на язык (`blog/en/` существовал). Применил тот же паттерн к главной
странице вместо того, чтобы тащить в одностраничное приложение клиентский роутинг (react-router),
который до сих пор был не нужен — 3 реальных статических entry point'а (`/`, `/en/`, `/vi/`),
каждый отдельный HTML-документ с собственными title/meta/OG/JSON-LD, все используют одно и то же
дерево React-компонентов, управляемое словарём локали. Настоящие URL, независимо индексируемые,
без хрупкости чисто клиентского роутинга.

## Что сделано
- `src/i18n/{types,ru,en,vi}.ts` — типизированные словари, весь текст вынесен из компонентов
  (nav, кнопки, заголовки, FAQ, футер, ErrorBoundary, лейблы сегментов колеса). Проверено grep'ом
  каждого изменённого файла на кириллицу вне комментариев — не найдено (только dev-комментарии
  остаются на русском, это уже существующая конвенция проекта).
- EN — естественный американский английский, не дословный перевод. VI — сохраняет
  устоявшиеся английские крипто/гемблинг-термины (KYC, Provably Fair, rakeback), как это принято
  в реальном вьетнамском крипто-контенте, вместо неестественных дословных эквивалентов.
- `LocaleContext.tsx` + `bootstrap.tsx` + `main.{ru,en,vi}.tsx` — общая логика монтирования.
  `ErrorBoundary` получает locale/словарь пропсами, не через контекст — он оборачивает приложение
  СНАРУЖИ провайдера и должен пережить даже падение самого провайдера.
- `LanguageSwitcher.tsx` — флаг-дропдаун, сохраняет выбор в localStorage (`zhelezo_locale`),
  ссылки на реальные URL (не клиентское переключение).
- Автоопределение при первом визите (`navigator.language` → `/en/` или `/vi/`, иначе остаётся RU)
  — новый инлайн-скрипт в корневом `index.html`, срабатывает только если нет сохранённого выбора.
  Новый CSP-хэш добавлен в `public/_headers`, проверено `verify-csp-hashes.mjs`.
- `links.ts`: `refLinkForLocale()` добавляет `subId2=lang-<locale>` ДО публикации страниц, не
  задним числом — применил собственное правило проекта из `Experts/Finance/SYSTEM.md` (клик без
  subId-метки безвозвратно теряет атрибуцию) на этот раз проактивно.
- hreflang (ru/en/vi/x-default) на всех 3 страницах, `sitemap.xml` дополнен `xhtml:link`
  alternates на всех 3 главных URL, локализованные canonical/OG/Twitter/Organization+WebSite
  JSON-LD (переведены, не просто скопированы с русского с заменой lang).

## Осознанно не сделано (не скрыто, явно зафиксировано)
- Privacy Policy/Terms остаются едиными RU-страницами для всех локалей — юридический текст
  требует точности, а не просто беглого перевода; спешить с 2 языками юридического текста в этом
  же проходе означало бы риск того самого недоделанного юридического контента, которого проект
  избегал раньше.
- Перевод статей блога на VI (и завершение EN за пределами 1 существующей пилотной статьи) —
  отдельная, большая задача по контенту, не взята в этот проход, чтобы не гнать 12
  статей-эквивалентов в ущерб качеству — тот же принцип "2 полные статьи лучше 5 недоделанных",
  уже задокументированный в `Experts/Business/SYSTEM.md`.

## Верификация (реально запускал приложение, не только tsc/build)
- `npm run build` — чисто, 0 ошибок TypeScript, все 3 entry point собираются.
- CSP-хэш guard — проходит (10/10 инлайн-скриптов совпадают с `public/_headers`).
- Реальная Playwright-сессия против локального preview-сервера (не просто curl): скриншоты всех
  3 языковых версий целиком, визуально подтверждён корректный язык без console errors; реальный
  клик по переключателю RU→EN→VI, подтверждена смена URL/title/localStorage на каждом шаге;
  подтверждено автоопределение (en-US→/en/, vi-VN→/vi/, ru-RU и неизвестная locale de-DE→
  остаётся RU); regression-проверка всех старых страниц (блог, privacy-policy, terms, обе новые
  статьи блога) — все 200.
- Найден и исправлен реальный баг accessibility в процессе тестирования: `aria-label`
  переключателя языка перекрывал accessible name словом "Язык", из-за чего скринридер не называл
  ТЕКУЩИЙ выбранный язык, хотя зрячие пользователи его видят. Исправлено на "Язык: Русский" и т.п.

разрешился сам за секунды при повторной проверке.
