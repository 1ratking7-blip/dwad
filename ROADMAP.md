## 1. Технический фундамент — сделано

- Полный SEO/CRO/perf/a11y/security аудит лендинга, HSTS, CVE-фиксы, code-splitting, non-blocking fonts, custom 404, ErrorBoundary + client-side error reporting.
- CSP на основе точных sha256-хэшей инлайн-скриптов, CI-guard (`scripts/verify-csp-hashes.mjs`) — забытый хэш валит билд, а не тихо ломает скрипт в проде.
- **Полная мультиязычность RU/EN/VI**: 3 статических entry point'а (`/`, `/en/`, `/vi/`), собственные словари (не машинный перевод "как есть"), hreflang на всех 30 страницах, sitemap.xml на 27 URL с alternates, языковой переключатель + автоопределение браузера + localStorage, subId-разметка на каждой ссылке. Верифицировано Playwright (не только `tsc`/build), не гостовый — реально кликано/проверено.
- Блог: 7 статей × 3 языка = 21 страница, BreadcrumbList+Article JSON-LD, полная перелинковка.
- Privacy/Terms переведены на все 3 языка (переведены по смыслу, не построчно).
- **Известный операционный риск, зафиксированный и не повторявшийся с 2026-07-20**: у Cloudflare Pages для этого проекта нет GitHub auto-deploy — деплой только вручную (`npm run deploy` из `Landing/`), с обязательной `curl`-проверкой живого домена после. `git push` сайт не деплоит.

## 2. Сделано в этом проходе (2026-07-21)

### Реальный технический аудит (не гадание — Lighthouse + прод)
- Прогнан настоящий Lighthouse (mobile, simulated throttling) против `https://www.zhelezo.space/`. **До**: Performance 42, LCP 6.1с, TBT 1480мс, Speed Index 5.5с. Accessibility/SEO уже были 100.
- **Найдена и исправлена реальная причина**: GTM (148 КБ) и Яндекс.Метрика (94 КБ + 13 КБ webvisor-скрипт) уже грузились `async`, но их *выполнение* конкурировало с главным потоком браузера прямо в окне отрисовки LCP-элемента (H1). Решение — official vendor queuing pattern: `gtag()`/`ym()`-стабы определяются сразу (это чистый push в массив, без сети), а фактическая загрузка `gtag.js`/`tag.js` отложена до `window.load` + `requestIdleCallback` (фоллбэк `setTimeout`). Ни одно событие не теряется — оба стаба гарантированно ставят вызовы в очередь и реплеят их после реальной загрузки скрипта; это стандартный паттерн, а не хак.
- Задеплоено на все 3 языковые версии (RU/EN/VI, идентичный код), CSP-хэши пересчитаны и провалидированы (`verify-csp-hashes.mjs`), curl-проверка на проде подтвердила выкладку.
- **После**: Performance **42 → 82**, LCP **6.1с → 3.5с**, TBT **1480мс → 80мс**, Speed Index 5.5с → 4.8с. Измерено повторным прогоном Lighthouse на живом сайте, не предположение.
- robots.txt — корректен (`Allow: /`, ссылка на sitemap). sitemap.xml — 200 OK. Иконки/манифест — все резолвятся (`icon-192.png`, `icon-512.png`, `site.webmanifest`, `apple-touch-icon.png`); `/favicon.ico` отдаёт 404, но это не проблема — в `<head>` объявлен `<link rel="icon" href="/favicon.svg">`, который современные браузеры используют вместо неявного запроса `/favicon.ico`. Не чиню — не стоит усилий.
- Best Practices застрял на 77 — единственная причина: сторонние cookie от Яндекс.Метрики (`bh`, `yabs-sid`), которые Chrome помечает как "third-party cookies" в рамках депрекации 3p-cookies. Это часть работы самого счётчика Метрики (session recording/webvisor), не баг кода сайта — если понадобится поднять этот балл, вариант один: выключить `webvisor:true` в инициализации Метрики, но это осознанный компромисс аналитики vs. Lighthouse-балла, не техдолг.

### Индексация — реальная, тревожная находка
- Дважды проверено через живой веб-поиск (`site:zhelezo.space` и точный `"zhelezo.space"`) — **домен не появляется в выдаче вообще**, ни разу, ни в каком контексте.
- Это совпадает с давно известным блокером: Google Search Console так и не подключена (нужен verification-код, который может получить только пользователь через свой Google-аккаунт — заготовка `<meta name="google-site-verification">` в `index.html` готова и закомментирована, ждёт кода).
- Bing (`msvalidate.01`) и Яндекс.Вебмастер (`yandex-verification`) verification-теги на сайте есть, но не проверено, была ли когда-либо реально отправлена sitemap через их панели (это тоже ручное действие в веб-интерфейсе, не то, что можно сделать через код).
- **Что нужно от пользователя** (единственный реальный блокер роста органического трафика прямо сейчас):
  1. Зайти на https://search.google.com/search-console, добавить `zhelezo.space` как домен-property, получить verification-код.
  2. Прислать код мне — я вставлю его в `index.html`, задеплою, вы подтвердите владение в панели GSC.
  3. В GSC: Sitemaps → отправить `https://www.zhelezo.space/sitemap.xml`; для главных страниц (`/`, `/en/`, `/vi/`, блог-индекс) — URL Inspection → Request Indexing вручную, это ускоряет первый краул на дни-недели вместо ожидания органического обнаружения.
  4. Аналогично проверить (если ещё не делали) Bing Webmaster Tools и Яндекс.Вебмастер — сайты уже верифицированы мета-тегом, но sitemap там тоже нужно явно отправить в панели, это не происходит само.
